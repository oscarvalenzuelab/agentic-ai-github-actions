The problem

You ship 9 dependencies. You inherit 120.

  • Every direct dependency pulls a tree of transitive ones — each a package you now effectively maintain.
  • Four risk surfaces, usually watched by nobody: known vulnerabilities, license obligations, maintainer abandonment, and — new — embedded AI components.
  • Real example from this repo: a dependency pinned as "fixed" in March had accumulated 22 new advisories by July, including a man-in-the-middle vector. Nobody was looking.

The approach

Evidence is scripted. Only the narrative is generated.

1 · Collect
Deterministic scripts gather SBOM, advisories, license verdicts, AIBOM, and repository metrics. Reproducible, auditable, no model involved.

2 · Assess
GitHub Models reads the evidence and writes the assessment: overall risk, critical findings, recommendations. One bounded inference per week.

3 · Remediate
An agentic workflow acts on the findings: bumps vulnerable dependencies and opens a single reviewed pull request. Optional, sandboxed, capped.

A compliance artifact is only defensible if the data underneath it is deterministic.

How it runs

One weekly workflow, ten steps

  1. Pull the SBOM from the dependency graph — full tree, exact versions
  2. Batch-query OSV.dev for known vulnerabilities
  3. Evaluate every license against project policy (OSPAC)
  4. Scan for AI components, emit the AIBOM (ai-finder)
  5. Collect repository metrics for each direct dependency
  6. Compute 0–100 health scores (activity, bus factor, releases)
  7. Feed all evidence to GitHub Models with the selected analysis prompt
  8. Fall back to rule-based analysis if the model is unavailable
  9. Compose one consolidated report; file it as a tracking issue
  10. Publish AIBOM, SBOM, license verdicts to stable release URLs

Every step degrades gracefully — the workflow always produces a report.

Closing the loop

The agent that fixes what the report finds

  • A GitHub Agentic Workflow triggers after each analysis: reads the evidence — OSV findings and OSPAC license verdicts — bumps vulnerable dependencies, regenerates the lockfile, verifies the audit improves, and flags license findings for OSPO follow-up.
  • GitHub Copilot is the agent's core engine — native to Actions, running an included model chosen automatically from the plan. One repository secret: a fine-grained Copilot PAT.
  • All writes pass through validated safe-outputs: exactly one pull request and one issue comment. The agent itself is read-only, sandboxed, behind an egress firewall.

Evidence pipeline recommends → agent proposes → human disposes.

Making it yours

Three files control everything

  • Prompts — analysis-prompts/*.md are the analysis personas: comprehensive, security-focused, maintainer-burnout, community-health, license-compliance. Add a file, add a workflow option — done. A strict JSON contract is appended automatically so output stays parseable.
  • Model — a workflow input; any model in the GitHub Models catalog. Default is a small, free one.
  • Subject — point it at your own repository: the dependency graph, prompts, and reports follow automatically. Schedules are two cron lines.
